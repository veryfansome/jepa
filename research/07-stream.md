# The `stream` Chunk: Single- vs Multi-Vector Token Layouts for a Sequence World Model

## Abstract

The `stream` chunk of terminal-JEPA decides how each command and observation in an interleaved `cmd,obs,cmd,obs,…` history becomes token(s) for the causal sequence model: one pooled vector per item, or several vectors exposing sub-parts (directory entries, file lines, typed roles). This is the classic single-vector-vs-multi-vector representation choice that separates dense bi-encoder retrieval (DPR, E5) from late-interaction models (ColBERT, ColPali), and it is governed by an information-bottleneck tension — a fixed-length pooled vector is a lossy sufficient statistic, while a set of vectors preserves addressable structure at higher cost. This report surveys the theoretical families behind that choice and grounds them in terminal-JEPA's own results: naive multi-vector layouts (line-index strips, role-canonical slots) *failed* their matched single-vector controls, while an **additive path-keyed** layout — keep the aggregate observation vector and *add* up to K path-prefixed entry segments — was the first multi-vector stream to beat its control at full budget, and did so only on an architecture with token attention, staying neutral on the fast-weights champion that has no mechanism to address the extra tokens. The consistent lesson, in both the retrieval literature and this project, is that extra vectors help only when (a) they carry a stable, queryable key and (b) the model can attend to them.

## 1. The role of the stream in this world model

terminal-JEPA is a JEPA-style latent world model: a causal transformer (or fast-weights recurrence) consumes a frozen-encoder embedding of an exploration history and, at each command position, predicts the *latent* of that command's resulting observation, scored by next-observation retrieval against hard same-verb foils. The `stream` chunk is the adapter between raw `(command, observation)` steps and the token tensor the architecture sees. Its contract (`chunks/stream/baseline_interleave.py`) is four functions: `collate` (emit `tok [B,L,D]`, `types [B,L]∈{cmd,obs}`, `key_pad`, the fixed single-vector target `tgt`, and `cmd_mask`); `extract_cmd_pred` (slice the prediction at each step's command token); `flatten_predictions` (produce per-step predictions in step order for the eval); and `leakage_ok` (a causality probe).

Three constraints make the stream a *representation-only* lever, never an eval lever. **The target and eval space are fixed:** every stream predicts the same single standardized mean-pooled `z_obs` per step, scored by the same foils and baselines, so the content-verb margin means what it meant in R4 — only *what the predictor reads* changes. **No future leakage is mandatory:** within a step the order is `cmd_i, obs_i, [sub-segments of obs_i]`, so every observation-derived token sits strictly after its own command and strictly before the next command; `leakage_ok` corrupts `obs_t` and asserts that command predictions at steps `≤ t` are bit-stable (a hard −∞ filter). **The layout interacts with the architecture:** a multi-vector stream inserts extra tokens (stride `2+K` in the additive path-keyed variant vs. stride `2` in the baseline), and those tokens are only useful if the trunk can *route information from them to the command position* — the crux of every finding below.

## 2. Theoretical frameworks for token representation

### 2.1 Single-vector pooling and the information bottleneck

The baseline stream is a **single-vector-per-item** design: each observation is mean-pooled into one 768-d vector, exactly as Sentence-BERT pools token embeddings into a fixed-length sentence vector [6] and as dense retrievers like DPR [3] and E5 [4] compress a passage into one embedding. Pooling is attractive — cheap, permutation-robust, a clean target space — but it is fundamentally a *bottleneck*. The information-bottleneck view of deep representation learning [12] frames a layer as trading compression against predictive sufficiency: a pooled vector is a minimal statistic only insofar as the downstream task needs nothing it discarded. Deep Sets [10] makes the compression explicit — any permutation-invariant set function factors as `ρ(Σ φ(x))`, i.e. a pool over element encodings — which is exactly a mean-pooled observation: order-invariant, but with all per-entry identity summed away.

The cost of that summing has a sharp theoretical statement. Weller et al. [13] prove that a single `d`-dimensional embedding can realize only a bounded number of distinct top-k relevant sets — the bound is tied to the sign-rank of the query-relevance matrix — so for any fixed `d` there exist combinations of relevant items that *no* single-vector query can carve out, regardless of training. Their LIMIT dataset shows state-of-the-art embedders failing linguistically trivial but geometrically impossible retrievals. For terminal-JEPA this is the precise reason a pooled `ls` observation can be a poor key: a directory's individual entries are exactly the "combinatorial subsets" a later `cd`/`cat` command needs to address, and a single mean-pool cannot keep them all separable.

### 2.2 Multi-vector representations and late interaction

The retrieval community's answer is **multi-vector / late interaction**. ColBERT [1] keeps one contextualized vector per token and scores a query against a document by MaxSim (sum over query tokens of the max similarity to any document token) — a Chamfer-style set matching that *delays* interaction until scoring, retaining fine-grained addressability while still precomputing document vectors offline. Late interaction is consistently more effective than single-vector DPR on hard and definitional queries because relevance is decided by *matching sub-parts*, not by one compressed summary; ColBERTv2 [2] shows this quality holds while cutting the (order-of-magnitude larger) storage via residual compression. ColPali [11] carries the identical idea to documents-as-images: 1,024 patch vectors per page, MaxSim against query tokens, "preserving patch-level granularity" rather than pooling — and it beats pooled-feature OCR pipelines. The through-line: a **set of keyed vectors** beats one pooled vector precisely when the useful signal lives in localizable sub-parts.

Between the extremes sit **Poly-encoders** [5]: a bi-encoder augmented with a small fixed number of learned "code" vectors that attend over the context, then attend to the query — cheaper than a full cross-encoder, more expressive than a single dot-product. This is the design space terminal-JEPA's stream navigates: how many vectors per item, and what does each one key on?

### 2.3 Granularity, addressing, and the need for something to attend to

Multi-vector layouts only pay off if the architecture can *address* the extra tokens. ColBERT's MaxSim and ColPali's patch matching are explicit attention-like operators over the vector set; Poly-encoders literally attend over their code vectors. The Perceiver [9] makes the dual point: it distills a large, variable-size input into a tight latent bottleneck via *iterative cross-attention*, so the number of latent tokens and the attention that reads them are co-designed — extra input positions are inert without a read mechanism. Flamingo's Perceiver Resampler [14] similarly turns variable visual features into a fixed set of tokens that the language model then cross-attends. The recurring principle: **granularity is only as useful as the model's ability to attend to it.** This is the direct theoretical anchor for terminal-JEPA's central finding — the same multi-vector stream helped an attention architecture and did nothing on a fast-weights recurrence, because the latter has no token-level attention to bind a command to the right observation sub-segment.

### 2.4 Typed / structured layouts for heterogeneous streams

An interleaved `cmd,obs` stream is heterogeneous, and sequence models encode item *type* explicitly. BERT [7] adds segment embeddings to distinguish sentence A from sentence B; terminal-JEPA's `types ∈ {cmd, obs}` and its per-arch `Embedding(2, d)` are the same device, and multi-vector streams keep every sub-segment typed as `obs` so the causal mask and role signal are preserved. Tokenization granularity is itself a design axis: ViT [8] shows a sequence model's behavior is set by how the raw signal is *cut into tokens* (16×16 patches), and that the transformer then attends across those tokens — the layout decision and the attention are inseparable. Flamingo [14] and Perceiver [9] extend this to arbitrarily interleaved modalities with fixed-size resampled token sets. For a shell world model the analogous "structured layout" question is whether to expose an observation as one vector, K role-typed vectors (state / names / body / signature), or K path-keyed entry vectors — each a different bet about what a future command will need to address.

## 3. Domain-specific findings

### 3.1 What retrieval tells us

The retrieval evidence is unambiguous about *direction*: entry-level / token-level matching (ColBERT, ColBERTv2, ColPali) beats single-vector pooling (DPR, E5, Sentence-BERT) on tasks where relevance rides on sub-parts, at the cost of storage and a matching operator. But it is equally clear about *preconditions*: the gains require (a) a stable key per vector (contextualized token, patch position) and (b) an explicit interaction operator (MaxSim, cross-attention). Poly-encoders show diminishing returns from *unkeyed* extra vectors, and the embedding-dimension lower bound [13] explains why single-vector methods have a hard ceiling that only multi-vector or higher-dimensional schemes escape.

### 3.2 What terminal-JEPA found (the arch-dependent story)

terminal-JEPA tested three multi-vector streams against matched single-vector controls, all holding the target/eval space fixed:

- **`mv_obs_k4`** (line-index strips): split each observation into K=4 contiguous line strips, *replacing* the aggregate vector. Result **0.5361 vs control 0.5386** — a loss. The strips were arbitrary line-*index* boundaries, so segment `k` of two observations held unrelated entries (no cross-observation alignment, no stable key), and replacing the aggregate forced the arch to relearn its whole information path for ~0 headroom.
- **`r7_role_multivec`** (role-canonical slots: state / names / body / type-signature): fixed the alignment problem by keying each slot to a semantic *role*. Result **0.5880 vs control 0.5885** — still a wash. The key was the role, not the *location* a command actually queries, and the aggregate was again replaced.
- **`r8_pathkey_entries`** (additive path-keyed): two structural fixes. (1) **Additive, not substitutive** — layout `[cmd_i, obs_i, p_i^1..p_i^K]`, stride `2+K`, keeping the untouched aggregate `z_obs` token so the baseline information path is preserved exactly (and, under the fast-weights arch, the delta-rule write still fires only on the aggregate obs token). (2) **Path-keyed** — each `p_i^k` is the frozen-e5 embedding of a segment whose text *begins with the absolute path* it describes (per-entry paths for `ls`, the file path for `cat`), encoded in the same `passage:`/`query:` space commands query, so the retrieval-tuned encoder itself binds content to location. This was the **first multi-vector stream to beat its control at full budget: +0.0015 inner / +0.0027 final** — but only on the hippocampal *attention* architecture. On the fast-weights champion it was **neutral**, exactly because that arch "has no token attention to address the segments with," and in a later planning context the path-key stream stayed neutral a third time.

The arch dependence is the headline, and it matches the theory in §2.3 precisely: the additive path-keyed segments are *keyed vectors* (path-prefixed, in the command's own embedding space), so a MaxSim-like attention can bind a `cd /etc` command to the `/etc` entry vector from an earlier `ls /` — but a delta-rule recurrence, which writes and reads associative memory at command/observation tokens without cross-token attention, has no operator to exploit them. The two earlier failures are equally consistent: line-index strips lacked a stable key (like ColBERT tokens with shuffled positions), and role slots keyed on role not location; both also *replaced* the aggregate, discarding the working single-vector path (an information-bottleneck regression) for structure the arch could not yet address.

## 4. Design guidance and open questions

**For future `stream` implementations:**
1. **Be additive, not substitutive.** Preserve the working aggregate vector and *add* sub-vectors; the aggregate is the proven bottleneck-optimal summary, and replacing it forces a full relearn for uncertain gain (the `mv_obs_k4`/`r7` failure mode).
2. **Key every sub-vector on what a future command will address** — here, absolute paths, in the command's own embedding space — so the frozen retrieval encoder does the content-to-location binding for free. Unkeyed or role-keyed segments underperform (§2.1, §3.2).
3. **Match the layout to the architecture's read mechanism.** Extra tokens help attention trunks and do nothing for pure recurrences without token attention. A multi-vector stream should be co-selected with an arch that has a MaxSim-/cross-attention-like operator, or paired with an explicit addressing head. This is the single most actionable lesson.
4. **Respect the causal contract and the fixed target.** Sub-segments must sit strictly between their own command and the next; `leakage_ok` must corrupt *all* segments of an observation and stay bit-stable; the per-step target remains the single `z_obs` so fitness stays honest.

**Open questions.** (a) Would a MaxSim-style *readout head* over path-keyed segments unlock the gain on the fast-weights champion, decoupling "multi-vector layout" from "attention trunk"? (b) Is there a middle path — a Poly-encoder-like handful of *learned* code vectors [5] or a Perceiver-resampled fixed set [9] — that gives addressable structure without the storage blow-up ColBERTv2 [2] had to compress away? (c) The embedding-dimension bound [13] suggests the single-vector target itself may cap retrievability of large directories; does raising K (or the target dimension) on `ls`-heavy systems move the content-verb margin, or only the storage cost? (d) Does path-keying help *planning* (addressing a goal directory's entries) even where it is neutral for one-step prediction?
